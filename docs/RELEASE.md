# Релиз и откат — QuickSave Bot

Чеклист перед продакшеном и действия при необходимости отката.

## Чеклист перед прод

- [ ] `npm ci && npm run lint && npm test` проходят локально и в CI.
- [ ] Staging: образ собран, контейнер поднят, [STAGING_SMOKE.md](STAGING_SMOKE.md) выполнен (HTTP + ручные шаги в Telegram).
- [ ] Секреты: `TELEGRAM_BOT_TOKEN` для **прод-бота** только в секретах хоста/оркестратора, не в git ([SECRETS.md](SECRETS.md)).
- [ ] `LOG_LEVEL` для прода обычно `info` или `warn`; не `debug` без необходимости.
- [ ] Лимиты: при необходимости скорректированы `RATE_LIMIT_*` и `MAX_SAVE_CONTENT_LENGTH` под ожидаемую нагрузку (см. `.env.example`).
- [ ] Бэкап: план копирования тома/файла SQLite перед релизом ([DEPLOY.md](DEPLOY.md)).
- [ ] Команда знает, кто принимает решение об откате и как связаться с хостингом.

## Процедура релиза (типично)

1. Смержить изменения в основную ветку, дождаться зелёного CI.
2. На сервере: `git pull`, `docker compose build`, `docker compose up -d` (или ваш пайплайн).
3. Проверить `/health`, `/ready`, затем сценарий из smoke / короткий ручной тест в Telegram.

## Откат

**Быстрый откат приложения**

- Если используется Docker с тегами образов: вернуть предыдущий тег в compose и `docker compose up -d`.
- Если деплой из git: `git checkout <previous-tag-or-commit>`, пересобрать и перезапустить.

**Данные**

- Новая версия могла изменить схему БД (миграции). При откате на старый бинарь убедитесь, что версия схемы совместима; при сомнении восстановите файл БД из бэкапа, сделанного **до** релиза.

**Секреты**

- При откате токен бота обычно не меняется; если меняли токен в BotFather — обновите переменные окружения на всех средах.

**Связь с пользователями**

- При длительном даунтайме при необходимости уведомите пользователей через ваш канал поддержки.
